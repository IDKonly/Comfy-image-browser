# 개발 및 유지보수 가이드 (Development Guide)

이 문서는 ComfyView 프로젝트에 기여하거나 코드를 수정할 때 준수해야 할 엔지니어링 표준과 작업 프로세스를 정의합니다.

## 1. 정밀한 외과적 수정 (Surgical Strike Principle)

코드는 자산(Asset)인 동시에 부채(Liability)입니다. 모든 수정은 최소한의 라인으로 최대의 효과를 내도록 설계되어야 합니다.

*   **최소 범위 수정**: 요청받은 문제를 해결하는 데 필요한 라인만 정확히 수정합니다. 인접 코드의 포맷팅(따옴표, 띄어쓰기 등)을 본인의 취향대로 수정하는 행위(Drive-by Refactoring)를 엄격히 금지합니다.
*   **하위 호환성**: 모든 코드 수정은 기존 기능과의 호환성을 우선시합니다. 대규모 리팩토링이 필요한 경우, 별도의 제안(Inquiry)을 통해 승인된 후 진행합니다.

---

## 2. 테스트 및 검증 (Test-First Loop)

ComfyView는 신뢰할 수 있는 소프트웨어를 지향합니다. 모든 로직 수정은 검증을 거쳐야 합니다.

*   **TDD 루프**: 특히 메타데이터 파싱이나 데이터베이스 로직의 경우, 실패하는 테스트(Red)를 먼저 작성하고 이를 통과하는 최소한의 코드(Green)를 작성한 뒤 전체 테스트(Regression)를 수행합니다.
*   **Rust Tests**: `cargo test`를 통해 백엔드 로직의 무결성을 확인합니다. 특히 `metadata.rs`의 경우 `test/` 폴더의 샘플 이미지를 활용하여 다양한 노드 지원 여부를 검증합니다.
*   **Build Check**: 모든 작업 완료 후에는 `npm run tauri dev` 또는 `build.bat`을 통해 컴파일 에러나 런타임 크래시가 발생하지 않는지 확인합니다.

---

## 3. 코드 스타일 및 명명 규칙 (Style Guide)

*   **Rust**: 공식 `cargo fmt` 스타일을 따르며, `match` 문을 활용한 안전한 에러 핸들링을 선호합니다. `unwrap()` 사용을 지양하고 `Result`나 `Option`을 적절히 반환하십시오.
*   **React/TypeScript**: 명확한 타입 정의와 시맨틱한 컴포넌트 구조를 유지합니다. 스타일링은 Tailwind CSS를 사용하여 유틸리티 클래스 위주로 작성하되, 복잡한 로직은 별도의 Hook으로 분리합니다.
*   **State**: 전역 상태는 Zustand 스토어(`useAppStore.ts`)에 집중시키고, 각 컴포넌트는 필요한 상태만 구독하여 렌더링 성능을 최적화합니다.

---

## 4. 문서화 원칙 (Documentation Rules)

*   **Permanent Preservation**: 기존 문서는 삭제하지 않고 `docs/archive/`로 이동하여 보존합니다.
*   **Max 3 Depth**: 문서는 최대 3단계 헤더(H1, H2, H3)까지만 허용합니다. 그 이상의 깊이가 필요할 경우 문서를 분리하십시오.
*   **Semantic Layer**: 정보의 중요도에 따라 계층적으로 서술하며, '무엇(What)'보다 **'왜(Why)'**를 설명하는 데 집중하십시오.
