# 시스템 아키텍처 (Architecture)

ComfyView는 AI 이미지 라이브러리가 기하급수적으로 증가함에 따라 발생하는 브라우징 성능 저하와 데이터 파편화 문제를 해결하기 위해 고안되었습니다. 이 문서는 프로젝트의 핵심 아키텍처 설계 배경과 데이터 흐름을 설명합니다.

## 1. 하이브리드 설계 철학 (Hybrid Architecture)

ComfyView는 사용자 인터페이스의 민첩성을 위해 **React**를, 고성능 데이터 처리와 파일 시스템 접근을 위해 **Rust**를 채택한 하이브리드 구조를 가집니다.

*   **Tauri v2**: Tauri를 통해 가벼운 바이너리를 유지하면서도 운영체제 레벨의 API에 안전하게 접근합니다. 특히 `v2`의 향상된 IPC(Inter-Process Communication)를 활용하여 대량의 이미지 경로와 메타데이터를 프론트엔드로 신속하게 전달합니다.
*   **Rust Backend**: 파일 스캐닝, 썸네일 생성, 프롬프트 압축 등 CPU 집약적인 작업은 Rust의 `Rayon` 라이브러리를 통해 멀티스레드로 처리됩니다. 이는 단일 스레드 브라우저 환경에서는 불가능한 수만 장 이미지의 실시간 처리를 가능케 합니다.

---

## 2. 데이터 영속성 및 인덱싱 (Data Persistence)

ComfyView는 디스크의 물리적 이미지를 매번 검색하는 대신, **SQLite**를 활용한 지능형 인덱싱 시스템을 구축했습니다.

*   **SQLite Indexing**: `rusqlite`를 사용하여 이미지의 경로, 메타데이터, 수정 시간(mtime)을 저장합니다. 사용자가 특정 폴더를 열 때, 데이터베이스는 mtime을 비교하여 변경된 파일만 선별적으로 다시 스캔함으로써 지연 시간(Latency)을 최소화합니다.
*   **Virtualization**: 프론트엔드에서는 `react-window`를 통해 리스트 가상화를 구현했습니다. 이는 수만 개의 이미지 컴포넌트를 메모리에 유지하는 대신, 현재 화면에 보이는 영역만 렌더링하여 60fps 이상의 부드러운 스크롤 성능을 보장합니다.

---

## 3. 상태 관리 전략 (State Management)

어플리케이션의 전역 상태는 **Zustand**를 통해 관리되며, 프론트엔드와 백엔드 사이의 데이터 동기화는 다음 원칙을 따릅니다.

*   **Single Source of Truth**: 모든 이미지 데이터의 마스터 데이터는 백엔드의 SQLite DB에 존재하며, 프론트엔드 스토어는 이를 캐싱하여 UI에 반영합니다.
## 4. 트러블슈팅: 데이터베이스 및 동기화 (Database & Sync)

시스템 운영 중 발생할 수 있는 주요 데이터 무결성 문제와 해결책은 다음과 같습니다.

*   **폴더 접두사 매칭 문제**: 재귀적 폴더 검색 시 `folder LIKE 'images%'`와 같은 단순 쿼리는 `images_backup`과 같은 의도하지 않은 폴더까지 매칭할 수 있습니다. 이를 방지하기 위해 `(folder = ?1 OR folder LIKE ?1 || '/%')` 구문을 사용하여 정확한 서브디렉토리 트리만 매칭하도록 보장합니다.
*   **경로 동기화**: 이미지 이동(Keep/Trash) 또는 삭제 작업 시 물리적 파일만 조작하고 DB를 업데이트하지 않으면 검색 결과에 깨진 아이콘이 나타날 수 있습니다. `file_ops.rs`의 모든 명령은 작업 직후 DB의 경로 정보를 동기화하며, 스캐너는 주기적으로 실재하지 않는 DB 항목을 정리(Cleanup)합니다.
*   **경로 정규화**: 윈도우와 유닉스 스타일의 경로 혼용으로 인한 중복 등록을 방지하기 위해, 모든 경로는 저장 전 슬래시(`/`)로 통일하고 끝부분의 슬래시를 제거하는 정규화 과정을 거칩니다.
